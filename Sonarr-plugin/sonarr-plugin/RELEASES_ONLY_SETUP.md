# 📦 Releases-Only Repository Setup

This document explains how the dual-repository system works for distributing **only built executables** without source code.

---

## 🎯 **The Problem**

You want to:
- ✅ Keep source code in one repo (`jose987654/sonarr-ext`)
- ✅ Publish releases in another repo (`jose987654/sonarr-plugin`)
- ✅ Only show the **built executable ZIP** to users
- ❌ NOT show source code or GitHub's auto-generated archives

---

## 🏗️ **The Solution**

### **Two Repositories:**

| Repository | Purpose | Contents |
|------------|---------|----------|
| **sonarr-ext** | Development | Full source code, development files |
| **sonarr-plugin** | Releases | Only release tags and uploaded ZIPs |

### **How It Works:**

1. **Develop** in `sonarr-ext` (source code repo)
2. **Build** the executable with `build.bat`
3. **Push code** to `sonarr-ext` (source stays there)
4. **Create release** on `sonarr-plugin` (releases-only repo)
5. **Upload only** the built ZIP file

---

## ⚙️ **Setup Instructions**

### **Step 1: Create Releases-Only Repository**

1. Go to: https://github.com/new
2. Repository name: `sonarr-plugin`
3. Description: `SonarrSeedr Plugin - Automated torrent downloads via Seedr`
4. **Important**: 
   - ✅ Make it public (so users can download)
   - ❌ Don't initialize with README (keep it empty!)
   - ❌ Don't add any files

The repo should remain **empty** except for:
- Release tags
- Uploaded ZIP files

### **Step 2: Configure Git Remotes**

In your `sonarr-ext` repo, you should have:

```bash
git remote -v
```

Should show:
```
origin  https://github.com/jose987654/sonarr-ext.git (fetch)
origin  https://github.com/jose987654/sonarr-ext.git (push)
```

**Note:** We don't add `sonarr-plugin` as a remote! The release scripts handle it via `--repo` flag.

### **Step 3: Release Scripts Are Configured**

The scripts (`release.bat`, `release.ps1`) are already configured to:

1. **Push code** to `origin` (sonarr-ext):
   ```bash
   git push origin main
   git push origin "v1.2.0"
   ```

2. **Create release** on `sonarr-plugin`:
   ```bash
   gh release create "v1.2.0" "SonarrSeedr-v1.2.0.zip" \
       --repo jose987654/sonarr-plugin
   ```

---

## ⚠️ **About GitHub's Auto-Generated Archives**

### **The Issue:**

GitHub **automatically** creates these for every tagged release:
- Source code (zip)
- Source code (tar.gz)

These **cannot be prevented** via the GitHub API or CLI.

### **The Solution:**

Since `sonarr-plugin` repo is **empty** (no source code), the auto-generated archives are:
- Empty or minimal (just the tag)
- Not useful to anyone
- **Clearly marked in release notes** to NOT download them

### **Release Notes Template:**

Every release now includes:

```markdown
## ⚠️ IMPORTANT: Download Instructions

**📥 DOWNLOAD THIS FILE ONLY:**
### `SonarrSeedr-v1.2.0-20250211_183835.zip`

**❌ DO NOT download "Source code (zip)" or "Source code (tar.gz)"**
Those are auto-generated by GitHub and are NOT the application!
```

This makes it **crystal clear** which file users should download!

## 🔄 **How Auto-Update Works**

The auto-update feature (`app/service/update_service.py`) is configured to:

1. **Check** for latest release on `sonarr-plugin` repo
2. **Find** assets that match `SonarrSeedr*.zip` (excludes source archives)
3. **Download** only the correct ZIP file
4. **Extract** and update automatically

### **Code That Ensures Correct File:**

```python
# Only look for SonarrSeedr*.zip (not source code archives)
for asset in release_data.get("assets", []):
    asset_name = asset.get("name", "")
    # Must start with "SonarrSeedr" and end with ".zip"
    if asset_name.startswith("SonarrSeedr") and asset_name.endswith(".zip"):
        download_url = asset.get("browser_download_url")
        break
```

This ensures users **never accidentally download** source code archives!

## 🆘 **Troubleshooting**

### **Q: Can I completely remove source code archives?**
**A:** No, GitHub always generates them for tagged releases. But since the repo is empty, they're not useful anyway.

### **Q: What if someone downloads the source archive?**
**A:** It will be empty or just contain the tag. Clear warnings in release notes prevent confusion.

### **Q: Can I use one repo for both?**
**A:** Yes, but then users see all your development files, issues, PRs, etc. This setup is cleaner for distribution.

### **Q: Do I need to maintain two repos?**
**A:** Only develop in `sonarr-ext`. The `sonarr-plugin` repo is automatically managed by the release scripts!
